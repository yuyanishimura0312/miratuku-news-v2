AIRR 補遺III 2026年5月3日

AIリサーチ・ルール基盤 [補遺III] — 技術実装詳細

Anthropic Claude Research の multi-agent アーキテクチャ詳解、Citation Agent への PaperQA2 風 Fulltext 取得層（Unpaywall + PDF parser + Claude 本文照合）の実装、ICML/ICLR/NeurIPS/CVPR/ACL 2025-2026 の査読 LLM ポリシーと検出技術。本ルール基盤の技術背景を一次ソースから詳説する。

CHAPTER 01

Anthropic Claude Research の Multi-Agent アーキテクチャ

Anthropic の Research システムは、単なる「検索を並列化した RAG」ではなく、Lead Researcher が探索計画を立て、複数 Subagent に独立した調査タスクを委任し、最後に Citation Agent が出典整合性を補強する非同期・高トークン消費型の agent orchestration である。公開情報から確認できるのは設計思想と一部の運用上の数値であり、内部プロンプト・厳密な message schema・Citation Agent のコードレベル実装・scheduler の詳細は未公開である。

1.1 Lead Researcher と Subagent の通信プロトコル

User query ↓ Lead Researcher - 問題分解 - 調査計画作成 - Subagent への委任 - 進捗統合 - 最終レポート生成 ↓ Subagents (独立した context window と tools) - 独立した検索・調査 - 必要ならさらに tool use - 要約・発見事項を Lead に返却 ↓ Lead Researcher - 結果統合 - 不足領域の追加探索 - 最終回答

通信プロトコルとして公開されているのは概念レベル。task-delegation protocol 的に運用されており、natural-language task contract + tool-result summarization に近い構造である。Lead が Subagent に「何を調べるか」を自然言語で渡し、Subagent が「何を見つけたか」を構造化要約として返す。peer-to-peer 通信や厳密な schema は公開情報では確認できない。

1.2 Citation Agent の役割

Citation Agent は、最終回答の信頼性を補強する後段 agent として説明されている。「検索 agent」ではなく、生成済みレポートと source material の対応付けを行う検証・引用補助 agent である。Citation Agent の prompt、citation span matching のアルゴリズム、embedding/lexical search の有無、chunking strategy、confidence threshold、hallucinated citation の reject 条件、code-level implementation はすべて未公開である。Anthropic の Citations API は document を渡すと回答内で該当箇所への citation を返せる機能だが、Research system 内の Citation Agent と Citations API が同一実装かは公開情報では確認できない。

1.3 並列度と token consumption の関係

公開された運用数値

Anthropic Engineering ブログによれば、multi-agent research system はチャット比 15× の token を消費するが、内部 Research eval で単一 Opus 4 比 +90.2% の精度を達成し、複雑クエリで時間を最大 90% 短縮した。

このトレードオフは、ルール基盤 R4-6（コスト管理）の設計に直接効く。並列度を上げれば精度は上がるが、token / $ / レイテンシのコストは線形以上で増える。重要案件（医療・法務）では多軌道検証を優先し、低リスク案件（PESTLE シグナル）では並列度を抑える運用判断が必要。

1.4 実運用上の落とし穴

  • Subagent 間の重複調査: Lead の問題分解が粗いと、複数 Subagent が同じソースを並列に取得してトークンを浪費する
  • Citation Agent の偽陽性: 書誌情報レベルの一致だけで citation を付与すると、本文が主張を支持していない場合がある（→ 補遺III §2 の PaperQA2 fulltext 層で対処）
  • Lead の bottleneck: 全 Subagent 結果が Lead に集約されるため、Lead の context window 圧迫が問題化する場合がある
  • 長期記憶の欠落: 公開情報からは Research system に永続メモリの統合は確認できない（Letta / Mem0 等の併用が必要な場面あり）

CHAPTER 02

Citation Agent への PaperQA2 風 Fulltext 取得層

本ルール基盤の Citation Agent は、書誌情報レベル（タイトル + 著者 + 年）の topical relevance までを Claude セマンティック検証で実装している。PaperQA2（Skarlinski et al. 2024, arXiv:2409.13740）が示すように、本格的な引用支持率測定には本文照合（fulltext-level support verification）が必要である。

2.1 設計

codex-team の調査により、以下の追加層を verify_citations.py に実装した。既存 API は変更せず、新メソッドを add する形式。

追加された関数（公開 API）

  • FullTextResult: PDF URL / 取得テキスト / parser / バイト数を保持するデータクラス
  • fetch_unpaywall_pdf_url(c): DOI から Unpaywall API で best_oa_location.url_for_pdf / oa_locations[].url_for_pdf を取得
  • fetch_full_text(c): DOI → Unpaywall → PDF download → text extraction の一気通貫
  • parse_pdf_text(pdf_bytes): pdfplumber 優先（レイアウト保持）、fallback で PyPDF2
  • build_full_text_support_prompt(...): Claude に「トピック一致」ではなく「本文が主張を直接支持するか」を JSON で返させるプロンプト

レート制限と PDF サイズ制限

RATE_LIMIT_PER_HOST["api.unpaywall.org"] = 1.0 # 1 req/sec (100K/day 制限) MAX_PDF_BYTES = 25 * 1024 * 1024 # 25 MB cap MAX_FULL_TEXT_CHARS = 120_000 # context window 配慮

2.2 設計判断

  • pdfplumber 優先: PyPDF2 と比べてレイアウト保持が比較的良く、本文照合に向く
  • 本文照合プロンプト: 「topical relevance（トピック関連）」ではなく「direct support（主張の直接支持）」を判定させる。これにより既存の semantic verification より厳しい基準で評価可能
  • Unpaywall 優先: 100K/day のクォータ内で OA 経由の PDF 取得を実現。paywall 越えは試みない（ルール R7-3 に整合）

2.3 期待される効果

既存の Claude セマンティック検証では本ルール基盤の自己検証で支持率 62.9% に留まっていた。本文照合層を加えることで、(a) 真の引用支持率の把握、(b) 偽陽性（書誌一致だが本文不一致）の検出、(c) 本文への direct quote の自動抽出が可能になる。これは PaperQA2 が precision 85.2±1.1%、accuracy 66.0±1.2% を達成した設計に追従する。

CHAPTER 03

査読プロセスでの LLM 使用 — 主要会議 2025-2026 ポリシー

Liang et al.（ICML 2024, arXiv:2403.07183）が査読本文の 6.5–16.9% に LLM 改変痕跡を確認したことを契機に、主要 ML/CS 会議は 2025-2026 にかけてポリシーと検出を急速に強化している。

3.1 各会議のポリシー比較

会議査読での LLM 使用原稿投入違反時
ICML 2026二重ポリシー: A=保守的(不使用)、B=許容的(理解+ポリッシュ)。査読者が事前選択privacy-compliant LLM のみ可（Policy B）flag + manual verify
ICLR 2026開示なき LLM 使用は即時 reject禁止desk reject
NeurIPS 2025査読者は submission/コードを LLM と共有禁止禁止明文化
CVPR 2025査読での LLM 使用を完全禁止禁止明文化
ACL 20262023 ポリシー継承: 文法補助・短文 paraphrase は許容、新規生成は明示開示原則禁止開示義務

3.2 検出技術: プロンプトインジェクション透かし

ICML 2025 は提出 PDF に invisible なプロンプトインジェクションを埋め込み、LLM がそれを実行すると検出可能なフレーズが生成される仕組みを実装した。これにより 506 名の査読者から 795 件（全体の約 1%）の AI 生成査読を flag、manual verify で false positive を排除した。

3.3 回避策・抜け穴・限界

  • 透かし回避: PDF→テキスト変換時に invisible 命令を除去すれば回避可能。OCR 経由なら命令文字列も除去される
  • local LLM: Ollama / llama.cpp 等のローカル LLM 使用は外部からは検出不可
  • 「校正のみ」の境界: 文法補助と新規生成の境界は曖昧で、運用は査読者の自己申告に依存
  • 検出閾値: false positive を避けるため manual verify 必須となり、スケールに限界
ルール基盤への含意

本ルール基盤の R7-5（LLM 検出ツールを一次根拠としない）は、透かし技術の限界と回避策の存在を踏まえると依然として正しい。査読者の自己申告と説明責任を中心とし、検出は補助に留めるべきである。

3.4 倫理的論点

  • プライバシー: 査読対象の未公開原稿を公開 LLM に投入することは守秘義務違反。これは ICML/ICLR/NeurIPS/CVPR/Nature/Science すべてに共通する基本ライン
  • 透明性ジレンマ: Schilke & Reimann（2025, OBHDP）は AI 使用開示が信頼を低下させることを実証。開示単独でなく検証プロセスとセットで提示する必要
  • 非英語話者・新人研究者の支援: 文法補助での LLM 使用を完全禁止すると、非英語話者や新人研究者を不利にする。境界設計は慎重を要する
