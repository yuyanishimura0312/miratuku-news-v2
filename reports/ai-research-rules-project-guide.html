AIRR 補遺II 2026年5月3日

AIリサーチ・ルール基盤 [補遺II] — プロジェクト別実装ガイド

miratuku（未来予測・PESTLE シグナル収集）/ foresight（動向調査・文献統合）/ esse-sense（概念分析・哲学的考察）の 3 プロジェクトに対し、ルール基盤の各原則をどう具体実装するかを、推奨アーキテクチャ・採用閾値・運用 Tips まで踏み込んでガイドする。

PROJECT 01

miratuku — 未来予測 / PESTLE シグナル収集

miratuku-news-v2
PESTLE analysis-based news platform v2
計画分離型 多軌道シグナル Citation Agent Trajectory Logger

既存の signals.html / js/pages/signals.js は ai_analysis.json の weak_signals を表示している。新規アプリを作るより、既存パイプラインの各ステップに検証層を差し込む方が ROI が高い。

推奨アーキテクチャ

Si, Yang, Hashimoto（ICLR 2025）が 100 名以上の NLP 専門家研究で示した「LLM の novelty は高いが feasibility は低い」を前提に、人間が feasibility で再選別する stage-wise human-in-the-loop を必須化する。AI co-Scientist（Gottweis 2025）の generate-debate-evolve 方式（tournament evolution）は miratuku のシグナル収集にそのまま転用できる。

シグナルメタデータ拡張（既存 ai_analysis.json への差分）

{ "signal": "...", "description": "...", "pestle_categories": ["Technological", "Legal"], "three_horizons": "H3", "cla_depth": "systemic", "composite_score": 7.4, "evidence": [ { "title": "...", "url": "...", "source_type": "news|paper|policy|dataset", "retrieved_at": "2026-05-02T00:00:00Z", "supports_claim": true } ], "verification": { "citation_existence": 1.0, "support_rate": 0.87, "self_consistency_k": 5, "semantic_entropy": 0.42, "abstained": false } }

重点ルール

  • R5-1 novelty/feasibility 前提（人間が feasibility 再選別）
  • R5-2 多サンプル多様性確保（temperature 高 + クラスタリング）
  • R3-1 多軌道検証（同一シグナルを 5 つの異なる視座で評価）
  • R6-2 採用ゲート（composite_score < 5 は abstain、citation_existence < 95% は再検索）

採用閾値（B1 テンプレ準拠）

  • 引用精度 D: ≥ 95%（PESTLE シグナルは公開ニュース中心のため標準）
  • 情報収集 B: BrowseComp ≥ 30%、GAIA Lv1 ≥ 70%
  • $/シグナル: < $0.30、レイテンシ p95 < 30 秒
PROJECT 02

foresight — 動向調査 / 文献統合

foresight-app
未来洞察・動向調査の文献統合プラットフォーム
PaperQA2 型 RAG Citation Agent (full gate) CoVe + Citation 6軸全測定

Ajith ら（LitSearch, EMNLP 2024）の実証研究に従えば、Google Search 単独は最良の密検索より recall@5 で 32 ポイント劣後する。foresight では Semantic Scholar / OpenAlex / arXiv / Google Scholar の 4 系統並列 + embedding 密検索 をデフォルトとする。

推奨アーキテクチャ

PaperQA2（Skarlinski 2024）は LitQA2 で precision 85.2%、生物学論文 1 編あたり 2.34±1.99 件の矛盾検出（70% が人間専門家により妥当判定）を達成した。foresight ではこの設計を踏襲し、Search → Gather Evidence → Answer の RAG エージェントに Citation Agent をパイプラインの必須ゲートとして統合する。

パイプライン

[文献検索] Semantic Scholar + OpenAlex + arXiv + Google Scholar ↓ (4 系統で取得した結果を embedding でクラスタ) [Evidence Gather] PaperQA2 風: 各論文の per-paper 要約と関連度評価 ↓ (Self-Consistency k=5 でエビデンス強度を評価) [Citation Agent] 全引用を Crossref/arXiv/OpenAlex で実在検証 ↓ (実在率 ≥ 95% を必須ゲート) [CoVe] Chain-of-Verification で長文事実性を確認 ↓ [出力] 引用必須 + Trajectory Logger ですべて記録

重点ルール

  • R2-1〜R2-6 引用関連（DOI/PMID/arXiv 検証 + Google 単独不可）
  • R6-1 6 軸を全測定（特に C 文献統合 軸）
  • R4-2 計画分離（計画フェーズを human-in-the-loop で承認）
  • R3-5 矛盾検出を独立タスクで実行
PROJECT 03

esse-sense — 概念分析 / 哲学的考察

esse-sense (-data-kit, -keyword-analysis, -support-report)
概念分析・哲学的考察 — LLM が苦手な領域
多軌道検証強化 バイアス点検 流暢さ ≠ 正しさ abstain 一級市民

esse-sense は LLM が構造的に苦手な領域である。Bender et al.（FAccT 2021）の「確率的オウム」批判が直撃する領域であり、ハイデガー / ベンヤミン / フーコーといった概念をモデルが「もっともらしく」扱える流暢さは、概念精度の保証ではない。

推奨アーキテクチャ：5 つの哲学的立場の並列実行

同一の概念質問を、異なる立場（例: 現象学的 / 構造主義的 / 分析哲学的 / 批判理論的 / 東洋思想的）から並列に解かせ、結果を集約する。これは Anthropic Multi-Agent Research の orchestrator-worker パターンに、立場ごとの subagent prompt で実装する。

立場別 subagent プロンプトの設計（例）

立場1: 現象学（フッサール、ハイデガー、メルロ＝ポンティ） 立場2: 構造主義・ポスト構造主義（レヴィ＝ストロース、フーコー、デリダ） 立場3: 分析哲学（ウィトゲンシュタイン、クワイン、ローティ） 立場4: 批判理論（フランクフルト学派、ベンヤミン、ハーバーマス） 立場5: 東洋思想（老荘、禅、京都学派） 各立場で同一概念を独立に解析 → Lead が統合 → 矛盾点を明示的に保持 （強制的に「合意」させない設計が重要）

abstain 判定の再定義

esse-sense では abstain（答えない）の閾値を意図的に低く設定する。Self-Consistency k=5 のうち多数決クラスタが 60% 以上を取れない、または 5 立場の解釈が顕著に分岐する場合は「LLM 解釈は提示しない」を選ぶ。代わりに「各立場での暫定解釈」と「分岐の所在」のみを提示し、結論は人間に委ねる。

バイアス点検の必須化

Algaba et al.（arXiv:2405.15739, 2024）は LLM の引用が高引用論文に偏る Matthew effect を統計的に立証した。esse-sense の文献選定では、(a) 西洋哲学優位、(b) 男性著者優位、(c) 英語圏文献優位 のバイアスが発生しやすい。各成果物に「バイアス点検レポート」（女性著者比率、非西洋哲学比率、第三世界文献比率等）を必須付帯する。

内製 50 問の哲学領域版

  • Lv1 (4 問): 用語の定義（哲学辞典準拠の確認問題）— "Dasein" の Heidegger 1927 における原義は何か等
  • Lv2 (4 問): 思想家間の比較（Foucault と Deleuze の権力概念の差異等）
  • Lv3 (2 問): 概念史の流れの再構成（hermeneutics のシュライアマハーからガダマーまでの変遷等）

重点ルール

  • R3-1 多軌道検証（5 立場並列、合意強制なし）
  • R7-6 バイアス点検（西洋・男性・英語圏優位の警告）
  • R7-1 流暢さ ≠ 正しさ（流暢な哲学的散文に騙されない）
  • R4-7 abstain を一級市民に（不明領域では人間にエスカレーション）
CHAPTER 04

プロジェクト横断の運用

3 プロジェクトとも以下を 必須インフラ として共有する。

  • Citation Agent（C2）を成果物パイプラインに必須ゲートとして組込
  • Trajectory Logger（B3）で全実行を JSON-Lines で記録
  • Verification Stack（C4）で重要判断を Self-Consistency + CoVe + Semantic Entropy で多層検証
  • Red-Team Agent（C5）を主要成果物に対して定期実行（偽引用混入・トピックドリフト・誘導質問）
  • 6 軸ダッシュボード（C3）で四半期スコアを可視化

運用ルールは miratuku → foresight → esse-sense の順に厳格化する。最も寛容な miratuku（PESTLE シグナル収集）でも citation_existence ≥ 95% は最低基準である。最も厳格な esse-sense（哲学的考察）では abstain 判定の閾値を下げ、5 立場並列を必須化する。
