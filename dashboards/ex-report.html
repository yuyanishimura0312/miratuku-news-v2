CHAPTER 02

品質改善の経緯 — 監査から補正まで

2026-05-06 に実施した第一次品質監査（読取専用）で、persons カノニカル化の崩壊・ministry_code の大小不整合・member_orgs の表記揺れという3件の Critical 問題が検出された。また、appointments の重複103グループ・council_members の重複664行・wayback_discovered の orphan 171件などの Major 問題も同定された。

これらを解消するため Q1-Q4 の4並列補正パイプラインを実装し、各 Track ごとに --inspect / --dry-run / --execute の三段階で進行した。実行前に data/experts.db.YYYYMMDD-HHMMSS.bak を自動作成し、migration_log テーブルへ実行履歴を永続記録する設計とした。

補正の哲学
削除よりも印付け（notes='garbage_candidate' 等）を優先することで、全補正をロールバック可能な状態で残した。Track A/B 本番実行時に再評価できる余地を確保している。

CHAPTER 04

Track A — Wayback 遡及（本番準備完了）

Wayback Machine の過去スナップショットから過去年度の任命データを正規化展開する Track A は、本番実装スクリプト scripts/track_a_wayback_run.py（1,003行）が完成し、dry-run 検証で 3/3 件の取得に成功している。MeCab + 姓辞書ハイブリッド名寄せ、3段階マッチング（完全一致 → alias → 姓+所属近似）、3経路振り分け（自動確定/レビュー/未マッチ候補）を実装。wayback_discovered.status='skipped' の 2,150件 + status='done' の 368件が処理対象である。

本番実行時の安全機構として、(1) デフォルト dry-run、(2) 1.5秒以上のレート制限、(3) X-Accept-Reduced-Priority ヘッダ、(4) 429リトライ指数バックオフ、(5) 10件ごとのバッチコミット、(6) BEFORE/AFTER スナップショット比較、を組み込んでいる。ロールバックは source_type='wayback' AND created_at >= '...' で巻き戻し可能。

python3 scripts/track_a_wayback_run.py --no-dry-run --status done --limit 50
CHAPTER 05

Track B — researchmap / ORCID 連携（PoC 完了）

研究者プロファイル拡充の Track B は、PoC（scripts/track_b_researchmap_poc.py）で東大所属の上位委員10人を実APIで照合した。researchmap permalink は 8/10（80%）の取得に成功し、仕様書のカバレッジ予測（rmap 70-80%）の上限に到達した。一方 ORCID は 0/10 で、「東京大学」直接渡しでは Solr マッチが成立しないことが判明した。

本番実装に向けて、(1) researchmap permalink ごとの https://api.researchmap.jp/{permalink} 二段呼び出しによる氏名・所属の正確な取得、(2) ORCID 検索の英語所属辞書化（東京大学 ↔ University of Tokyo / Tokyo University）、(3) ローマ字変換併用、の3点が必要である。Q3 の member_orgs 正規化が完了したことで、所属組織キーワードによる二段階照合の精度が改善する。

CHAPTER 06

Track C — GTAテーマ精緻化（設計完了）

GTAテーマ分類の精緻化は、現状647サブテーマのうち56%（362件）が1件のみの紐付けという過度な細分化を解消することが中心課題である。設計仕様書（docs/track_c_gta_refinement_spec.md）では、(1) 多重分類スキーマへの移行、(2) 統合・属性化による粒度改善（647→420-450件）、(3) クロスドメイン検出、(4) 未分類14審議会の再分類、を5段階の Phase 計画として整理した。

未分類14件は中東情勢・米国関税措置・監査法人ガバナンス等の時事的・横断的性質を持ち、既存43中核テーマのどれにも収まらない。「国際関係・外交戦略」「金融・企業統治」等の新規中核テーマの追加が推奨される。

CHAPTER 07

Track D — PD完全リンク × パネルデータ化（設計完了）

姉妹DB Policy DB（policy-db-new、30,118事業）との完全リンクは、council_project_links（25,736件、信頼度3段階）を介して実現する。重要な発見として、PD側に councils テーブルが存在せず、両DBの結合には ATTACH DATABASE が必須である。council_project_links の信頼度は、組織マッチ44件・キーワード11,384件・名称14,308件の3層構造を持つ。

2021-2025年度の5年不均衡パネル（審議会数134-145件/年、関連人物数2,356-2,482人/年）として整備し、(1) 人物×予算マトリクス、(2) 府省横断プレイヤーTOP100、(3) 人事×予算連動、(4) HHI集中度、の4分析クエリを実行可能にする。新規6コンポジットインデックスで最も重いクエリ4でも2-4秒以内を達成する設計。

CHAPTER 08

拡張ロードマップとリスク

第二期拡張は3フェーズで進行する。Phase 1（即時着手）では Track A 本番投入と Track B 本番実装を並列実行し、Phase 2（1ヶ月）で Track C スキーマ移行と Track D インデックス追加・パネル化、Phase 3（2-3ヶ月）でクロスDB可視化レイヤーを公開する。

主なリスクは、(1) Wayback の rate limit / IP制限による Track A の長期化、(2) researchmap の SPA 化による Track B のスクレイピング失敗、(3) PD側のスキーマ変更による Track D の互換性破綻、の3点である。それぞれ、(1) X-Accept-Reduced-Priority + バックオフ、(2) API 二段呼び出しへの切替、(3) migration_log 同等の検出機構、で緩和する。
